# Source-to-model attestation v1

Status: experimental deterministic build-provenance mechanism for the pinned
public washing-controller workflow. It is not a general RTL equivalence proof or
an independent assessment.

## Problem

Proof-carrying controller artifacts bind the supplied RTL source digest and
independently verify the supplied AIGER relation. Those checks do not establish
that the AIGER file was actually derived from the named source. Without that
link, a valid certificate could accurately verify the wrong compiled model.

## Contract

`source_model_provenance_manifest_version=1` is canonical, ordered LF text. It
binds:

- an exact 40-character Yosys revision;
- one to 64 ordered source/recipe/model triples;
- a normalized relative working directory for each triple; and
- completion with no trailing fields.

The public manifest is
[`corpus/rtl/wmcontroller/source-model-provenance-v1.txt`](../corpus/rtl/wmcontroller/source-model-provenance-v1.txt).

`scripts/attest-source-model-provenance.sh` rejects symlinks anywhere in the
source tree, more than 4,096 files, trees above 256 MiB, oversized or
noncanonical manifests, path traversal, unsupported tools, revision drift,
missing files, output overwrite, synthesis timeout, and regenerated byte drift.
It copies the bounded source tree to a private temporary directory, runs each
recipe with a 120-second deadline, and on Linux applies a 1 GiB address-space
limit, 64 MiB file-size limit, and 64-process limit before comparing the
regenerated AIGER bytes. The original tree is never modified.

Success emits a deterministic CSV containing the source, recipe, retained model,
and regenerated model SHA-256 digests. A row is attested only when both model
digests are identical. Timing, host paths, and compiler branding are excluded
from the artifact.

## Evidence

The retained public result is
[`results/public-washing-source-model-attestation-v1.csv`](../results/public-washing-source-model-attestation-v1.csv).
It proves byte-for-byte regeneration of:

1. the upstream washing-controller source into the controller AIGER model; and
2. the repository-authored physical-plant source into its AIGER model.

`scripts/test-source-model-provenance.sh` regenerates the result and requires byte
equality. Its controls reject retained-model drift, path traversal, wrong Yosys
revision, CRLF, and NUL. CI runs the test with the same maintained Yosys revision
recorded by the provenance manifest. The hosted workflow builds that exact
revision without ABC rather than trusting the runner's moving package version.

## Bound portfolio verification

The governed proof portfolio can require this evidence as part of the same
invocation:

```text
guarded-continuation-checker \
  verify-controller-proof-mtbdd-portfolio-resources-attested \
  QUERY.txt POLICY.txt ARTIFACT.proof-mtbdd-portfolio \
  PROVENANCE.txt ATTESTATION.csv
```

The command requires the first provenance subject to match the query's exact
controller source and model paths. Each distinct plant source/model pair in the
query must then appear exactly once and in first-use order. The verifier reads
and hashes every source, recipe, and model, verifies the canonical CSV through
the public `source_model_attestation` Rust API, and checks that its tool identity
matches the provenance manifest before checking the proof portfolio. Failure
produces no SAFE or UNSAFE answer. The typed
`ControllerProofMtbddPortfolioTool::verify_attested` API applies the same
contract without shell invocation.

## Claim boundary

For the attested command, this closes the substitution gap between the query,
the retained model bytes, and the proof portfolio. For the retained public
evidence, it covers these two pinned models only. It does
not prove Yosys correct, establish semantic equivalence independently of Yosys,
validate the physical fidelity of the repository-authored plant, or attest an
external partner model. General self-service projects still need a signed build
provenance format or reproducible recipe that a verifier can execute under an
approved toolchain.

The harness executes synthesis recipes and therefore accepts only trusted,
reviewed project inputs. It is not a service-side sandbox for hostile RTL or
Yosys scripts. Untrusted partner submissions must use GCC's separate isolated
evaluation profile or an equivalent network-disabled container.
